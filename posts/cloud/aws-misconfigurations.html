Do cấu hình sai (misconfiguration)

Sau khi tìm hiểu, mình đã tổng hợp được 10 cấu hình sai (misconfig) "đụng là thấy" trong môi trường AWS, vì sao nguy hiểm, dấu hiệu nhận biết, cách tự kiểm nhanh (console/CLI), và hướng khắc phục bền vững (chuẩn hóa bằng AWS Config/Security Hub/IaC).

1. S3 để lộ công khai (không chặn public, bucket policy hở, quên dùng OAC với CloudFront)

🔴 CRITICAL

Mức độ: Nghiêm trọng

Vì sao nguy hiểm:

Dữ liệu tệp sao lưu, log, ảnh khách hàng… dễ bị thu thập hàng loạt khi Block Public Access bị tắt hoặc bucket policy cho phép Principal:"*". Dù AWS đã bật mặc định Block Public Access và tắt ACL cho bucket mới từ 04/2023, nhiều hệ thống cũ vẫn còn cấu hình mở hoặc vô tình tắt đi.

Dấu hiệu nhận biết:

  • IsPublic: true ở bucket
  • CloudFront truy cập trực tiếp S3 origin (không dùng Origin Access Control – OAC)

Tự kiểm nhanh (CLI):

aws s3api get-bucket-policy-status --bucket <BUCKET>
aws s3api get-public-access-block --bucket <BUCKET>

Khắc phục bền vững:

  • Bật/duy trì Block Public Access ở account level
  • Dùng OAC khi phân phối qua CloudFront
  • Giữ Object Ownership = Bucket owner enforced để vô hiệu ACL
  • Thiết lập Security Hub theo FSBP (S3 controls) để cảnh báo lệch chuẩn

2. IAM quá rộng quyền & dùng root/long‑lived keys

🔴 CRITICAL

Mức độ: Nghiêm trọng

Vì sao nguy hiểm:

Quyền *:* tạo "đường cao tốc" cho leo thang đặc quyền; dùng root và không bật MFA là rủi ro chí mạng. AWS khuyến nghị chỉ dùng role, ít nhất‑đủ dùng, và MFA cho root.

Dấu hiệu nhận biết:

  • User đính kèm policy trực tiếp
  • Access key sống lâu
  • Root đăng nhập thường xuyên, không có MFA

Tự kiểm nhanh:

aws iam get-account-summary --query 'SummaryMap.AccountMFAEnabled'

Khắc phục bền vững:

  • Chuyển sang federation/SSO + role
  • Tách nhiệm vụ, bật MFA cho root
  • Audit wildcard bằng Access Analyzer
  • Theo dõi IAM controls trong Security Hub (FSBP)

3. Security Group/NACL mở 0.0.0.0/0 cho SSH/RDP/DB

🟠 HIGH

Mức độ: Cao

Vì sao nguy hiểm:

Cổng 22/3389/3306/5432… mở Internet = bề mặt tấn công trực tiếp.

Dấu hiệu nhận biết:

  • Inbound 0.0.0.0/0 vào cổng quản trị/cơ sở dữ liệu

Tự kiểm nhanh:

aws ec2 describe-security-groups \
  --filters Name=ip-permission.cidr,Values=0.0.0.0/0

Khắc phục bền vững:

  • Chỉ cho phép IP/ASN tin cậy (hoặc qua bastion/VPN/SSM Session Manager)
  • Dùng SG referencing giữa tầng
  • Bật Security Hub EC2.18/EC2.19/EC2.21 để bắt config nguy cơ cao

4. CloudTrail chưa chuẩn (không multi‑Region, không log validation, không đẩy CloudWatch)

🟠 HIGH

Mức độ: Cao

Vì sao nguy hiểm:

Thiếu dấu vết phá án; dễ bị anti‑forensic.

Dấu hiệu nhận biết:

  • Chỉ có trail ở một Region
  • Log file validation tắt
  • Chưa tích hợp CloudWatch Logs

Tự kiểm nhanh:

Vào CloudTrail → Trails xem Is multi‑Region và Log file validation.

Khắc phục bền vững:

  • Tạo 1 multi‑Region trail cho management events (read/write)
  • Bật validation + SSE‑KMS
  • Tích hợp CloudWatch Logs
  • Theo dõi CloudTrail.1/.2/.4/.5 trong Security Hub

5. GuardDuty & Security Hub chưa bật "toàn cơ quan"

🟡 MEDIUM

Mức độ: Trung bình

Vì sao nguy hiểm:

Thiếu phát hiện sớm các hành vi bất thường (crypto‑mining, credential exfil, DNS lạ…), thiếu bức tranh posture.

Dấu hiệu nhận biết:

  • Không có detector ở các Region
  • Security Hub chưa kích hoạt FSBP

Tự kiểm nhanh (GuardDuty):

aws guardduty list-detectors

Khắc phục bền vững:

  • Chỉ định Delegated Admin qua AWS Organizations
  • Auto‑enable GuardDuty & Security Hub cho mọi account/Region
  • Ingest findings về một chỗ để xử lý

6. Snapshot công khai (EBS/RDS) hoặc không chặn public sharing

🔴 CRITICAL

Mức độ: Nghiêm trọng

Vì sao nguy hiểm:

Snapshot = ảnh chụp dữ liệu/volume đầy đủ; public = ai cũng copy/restore được. RDS snapshot public cho phép mọi tài khoản tạo instance từ snapshot đó.

Dấu hiệu nhận biết:

  • EBS Block Public Access for Snapshots chưa bật
  • Có RDS snapshot bị share public

Tự kiểm nhanh (EBS):

aws ec2 get-snapshot-block-public-access-state

Khắc phục bền vững:

  • Bật Block Public Access for EBS Snapshots ở mọi Region
  • Dùng AWS Config rule `rds-snapshots-public-prohibited`
  • Đặt SCP/ci‑policy cấm public sharing

7. Mã hóa dữ liệu chưa bật mặc định (EBS/S3/RDS)

🟠 HIGH

Mức độ: Cao

Vì sao nguy hiểm:

Mất thiết bị/ảnh chụp/log = lộ dữ liệu nếu không mã hóa.

Dấu hiệu nhận biết:

  • EBS default encryption tắt
  • RDS chưa bật Encryption at rest
  • (S3 hiện mặc định mã hóa SSE‑S3 cho mọi bucket/đối tượng mới nhưng vẫn nên kiểm chứng và dùng KMS khi cần)

Tự kiểm nhanh:

aws ec2 get-ebs-encryption-by-default

Khắc phục bền vững:

  • Bật EBS default encryption
  • Tạo RDS với encryption (KMS CMK theo miền dữ liệu)
  • Xem xét chuyển S3 sang SSE‑KMS khi cần kiểm soát khóa/giám sát truy cập

8. EKS lộ API endpoint & thiếu audit logging/secret encryption

🟠 HIGH

Mức độ: Cao

Vì sao nguy hiểm:

API server public không giới hạn CIDR dễ bị dò quét; không audit log khó điều tra; K8s secrets không bật KMS envelope encryption = rủi ro lộ bí mật.

Dấu hiệu nhận biết:

  • endpointPublicAccess=true và publicAccessCidrs = 0.0.0.0/0
  • Control plane logs tắt

Tự kiểm nhanh (CLI):

aws eks describe-cluster --name <CLUSTER> \
  --query 'cluster.resourcesVpcConfig.{public:endpointPublicAccess,cidrs:publicAccessCidrs}'

Khắc phục bền vững:

  • Private endpoint (hoặc public + whitelist CIDR)
  • Bật control plane logging (API/Audit)
  • KMS secrets encryption cho EKS
  • Theo dõi EKS.1/EKS.3/EKS.8 trong Security Hub

9. API Gateway không gắn cơ chế xác thực (authorizer)

🟡 MEDIUM

Mức độ: Trung bình

Vì sao nguy hiểm:

Route "trần" = mọi người gọi được, kể cả tài nguyên nội bộ.

Dấu hiệu nhận biết:

  • Nhiều route NONE/UNSPECIFIED

Khắc phục bền vững:

  • Buộc JWT/Cognito/Lambda Authorizer
  • Gắn WAF nếu public
  • Bật logging/X‑Ray
  • Theo dõi APIGateway.8 (và .1/.4/.9) trong Security Hub

10. Thiếu "giám sát mạng" & rào chắn egress (VPC Flow Logs, VPC Endpoint, S3 VPC‑only)

🟡 MEDIUM

Mức độ: Trung bình

Vì sao nguy hiểm:

Không có VPC Flow Logs = mù lưu lượng; không ràng buộc VPC Endpoint = dữ liệu đi Internet public; một số dịch vụ managed (OpenSearch/RDS/ElastiCache…) để public endpoint.

Khắc phục bền vững:

  • Bật VPC Flow Logs cho mọi VPC
  • Dùng VPC Endpoint và bucket policy để chỉ cho phép truy cập S3 qua endpoint
  • Với OpenSearch, triển khai trong VPC, hạn chế truy cập bằng SG/IAM/FGAC
  • Theo dõi EC2.6/EC2.10 và OpenSearch in VPC trong best practices

Phụ lục: Mỗi mục kiểm thế nào cho "nhanh – gọn – chắc"

🔧 Bật chuẩn FSBP trong Security Hub

Để có bộ kiểm nền tảng (EBS public snapshots, SG mở, IMDSv2, EKS audit logging…). Đây là "bản đồ" misconfig chính thức từ AWS.

🛡️ Triển khai GuardDuty toàn tổ chức

Để có tín hiệu đe doạ thời gian thực (DNS lạ, credential gọi API bất thường…).

🔍 Quét nhanh bằng Prowler (open‑source)

Chạy "khám tổng quát" multi‑account, xuất báo cáo CSV/HTML để ưu tiên xử lý. Có hướng dẫn chính thức của AWS cho mô hình hợp nhất báo cáo Prowler.

9%

Nhiều tổ chức

Vẫn gặp sự cố vì misconfig – luận điểm "phần lớn lỗi cloud do khách hàng" vẫn được các hãng lớn nhắc lại đến 2025.

Kết luận

Qua quá trình tìm hiểu về AWS misconfiguration, mình nhận ra rằng không phải "nguy cơ Zero‑day" mới làm ta mất ngủ; một Security Group mở 0.0.0.0/0 hay một snapshot công khai đủ khiến đêm dài. Mình nghĩ cách tốt nhất là bắt đầu từ những thứ đơn giản – đo được – tự động hoá được: bật Security Hub (FSBP), GuardDuty toàn tổ chức, quét Prowler định kỳ, và đưa các rule "không mặc cả" (encryption, public‑access block, IMDSv2…) vào IaC/CI.
